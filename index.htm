Методические рекомендации

1. Жизненный цикл программного обеспечения (Software Development Life Cycle, SDLC)

1.1. Понятие и цель

Жизненный цикл программного обеспечения — это совокупность процессов, охватывающих все стадии существования программного продукта: от идеи до вывода из эксплуатации.

Цель — систематизация разработки, обеспечение управляемости, прогнозируемости сроков и качества продукта.

1.2. Основные этапы SDLC

ЭтапЦельОсновные результаты
1. Анализ требованийВыявление потребностей пользователей и постановка задачТехническое задание (ТЗ), спецификации требований
2. ПроектированиеРазработка архитектуры и структуры системыДиаграммы, модели данных, описание модулей
3. РеализацияПрограммирование и сборка системыИсходный код, бинарные файлы
4. ТестированиеПроверка корректности и соответствия ТЗТест-кейсы, отчёт о тестировании
5. Внедрение и сопровождениеРазвёртывание, обучение, обновленияРуководство пользователя, эксплуатационные документы

1.3. Классические и современные модели жизненного цикла

  • Каскадная (Waterfall) — строгая последовательность этапов, без возврата.
  • Итеративная / Инкрементальная — создание продукта по частям (итерациям).
  • Спиральная — цикл разработки с анализом рисков и уточнением требований.
  • Agile / Scrum / Kanban — гибкие методологии, короткие спринты, постоянная обратная связь.

1.4. Пример визуализации итеративного подхода

@startuml
start
:Сбор требований;
:Разработка архитектуры;
repeat
  :Реализация итерации (спринта);
  :Тестирование функционала;
  :Интеграция и демонстрация;
repeat while (Требуется доработка?) is (да)
:Подготовка релиза;
stop
@enduml

1.5. Принципы модульной разработки

  • Разделение системы на независимые модули с минимальными связями.
  • Каждый модуль выполняет одну логическую функцию (Single Responsibility Principle).
  • Повторное использование и тестируемость компонентов.
  • Изоляция ошибок и лёгкость модификации.

2. Принципы проектирования информационных систем

2.1. Многоуровневая архитектура

Современные приложения строятся по принципу трёхуровневой архитектуры:

  1. Уровень данных (Data Layer) — базы данных, хранилища, файловые системы.
  2. Бизнес-логика (Business Layer) — обработка данных, расчёты, правила.
  3. Представление (Presentation Layer) — интерфейс пользователя, API, отчёты.
@startuml
package "Presentation" {
  [Web UI]
  [Mobile App]
}
package "Business Logic" {
  [Service Layer]
  [Controllers]
}
package "Data Access" {
  [Repositories]
  [Database]
}

[Web UI] --> [Service Layer]
[Mobile App] --> [Service Layer]
[Service Layer] --> [Repositories]
[Repositories] --> [Database]
@enduml

2.2. Архитектура клиент–серверных приложений

  • Клиент инициирует запросы (например, через браузер или мобильное приложение).
  • Сервер обрабатывает запросы и возвращает ответы в виде JSON, XML и др.
  • Компоненты могут быть распределены по различным машинам.
PlantUML (component):
@startuml
actor User
node "Web Client" {
  component "Browser UI"
}
node "Server" {
  component "API Controller"
  component "Service Layer"
}
database "DB"

User --> "Browser UI"
"Browser UI" --> "API Controller" : HTTP/HTTPS
"API Controller" --> "Service Layer"
"Service Layer" --> DB : SQL / ORM
@enduml

2.3. Интерфейсы взаимодействия модулей

  • Все модули должны обмениваться данными через интерфейсы и контракты.
  • При изменении одного модуля остальные остаются работоспособными.
  • Пример: IDataModule, ILogicModule, IReportModule.

7. Методы тестирования

7.1. Классификация тестов

  • Модульное (Unit) — проверка отдельных функций или классов.
  • Интеграционное — взаимодействие модулей.
  • Функциональное — проверка пользовательских сценариев.
  • Регрессионное — повторное тестирование после изменений.
  • Нагрузочное — проверка производительности.

9. Экономическая эффективность информационной системы

9.1. Основные формулы

ROI (окупаемость инвестиций)

\[ \mathrm{ROI} = \frac{\text{Доход} - \text{Затраты}}{\text{Затраты}} \times 100\%\, \]

NPV (чистая текущая стоимость)

\[ \mathrm{NPV} = \sum_{t=1}^{n} \frac{CF_t}{(1+r)^t} - I_0\,, \]

где \(CF_t\) — денежный поток в период \(t\), \(r\) — ставка дисконтирования, \(I_0\) — первоначальные инвестиции.

PP (Payback Period) — срок окупаемости

Для равномерных потоков: \(\mathrm{PP} = \dfrac{I_0}{\overline{CF}}\).
Для неравномерных — минимальный \(T\), при котором кумулятивный дисконтированный поток \(\sum_{t=1}^{T}\dfrac{CF_t}{(1+r)^t} \ge I_0\).

9.2. Пример расчёта ROI и NPV

Инвестиции500 000 ₽
Доход (год 1)220 000 ₽
Доход (год 2)300 000 ₽
Доход (год 3)350 000 ₽
Ставка r10%

NPV:

\[ \mathrm{NPV} = \frac{220\,000}{1.1} + \frac{300\,000}{1.1^{2}} + \frac{350\,000}{1.1^{3}} - 500\,000 \approx 204\,000 + 247\,000 + 263\,000 - 500\,000 = 214\,000\,. \]

\(\mathrm{NPV} > 0\) ⇒ проект экономически целесообразен.

ROI:

\[ \mathrm{ROI} = \frac{870\,000 - 500\,000}{500\,000} \times 100\% = 74\%\,. \]

Срок окупаемости: проект окупается примерно к середине второго года.
